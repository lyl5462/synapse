# 研发会议室：节点人机协作流程（落地说明）

> 本文档沉淀需求澄清等 SOP 节点的**流程设计**与**实现约定**，与《多智能体研发会议室实现方案》互补。  
> 代码锚点：`src/synapse/rd_meeting/`（会议室通用规范已内嵌于 `room_skill.py::_MEETING_ROOM_RULES`）、`skills/whalecloud-dev-tool-ask-user/`。

---

## 1. 开会与节点执行

| 约定 | 说明 |
|------|------|
| **默认自动运行** | `open_meeting` 完成主控提示词组装后即 `schedule_run_node` 执行当前节点（无需 `auto_run_first_node` 参数） |
| **节点一次调度** | `MeetingRoomOrchestrator.run_current_node` 触发 Host **单次** `execute_task_from_message`；Host 在 ReAct 内可多轮工具调用 |
| **推进** | 智能体按 NODE_OUTPUTS 约定文件名落盘后，经校验（及 `human_confirm` 时人工 approve）由 `on_node_complete` 推进 `dev.status.current_node_id` |

---

## 2. 技能 vs 强制注入上下文

| 类型 | 来源 | 用途 |
|------|------|------|
| **Agent 技能** | 系统智能体管理 → `AgentProfile.skills` | 需求澄清、代码分析、文档生成等可执行能力（工具 + SKILL.md） |
| **强制注入（非技能）** | 会议室装配 | 会议规范、ask-user 问卷规范（Host）、能力卡片、会议目标、`human_confirm` 语义、工单/系统参数、`prompt_supplement` |

**说明**：配置模型中的 `node_overrides.skill_ids` 为历史字段，**不作为**节点技能来源；技能仅在 Agents 配置绑定。节点 binding 使用：`host_profile_id`、`worker_profile_ids`、`human_confirm`、`node_intent`、`hitl_form_schema`、`prompt_supplement` 等。

注入位置：

- **System**：`build_room_skill_prompt` → `agent._custom_prompt_suffix`（含内嵌的会议室通用规范 + Host 追加 `whalecloud-dev-tool-ask-user`）
- **User**：`build_node_prompt`（议程、会议目标、Worker 列表、运行时参数、人工确认说明、运营补充）

## 3. 人工介入四类场景

**统一 UI 通道**：主控在回复末尾输出 `whalecloud-dev-tool-ask-user` 规定的 `<!-- hitl-questionnaire -->` … JSON … 标记 → 后端 `extract_hitl_from_agent_output` → `room_state.status=human_intervention` + `MeetingHitlForm` 渲染。

> 通用对话的 `ask_user` **工具**与会议室表单未打通；会议室场景只认 **ask-user 技能 + 问卷 JSON**，不叫 IM tool。

| 场景 | 触发 | 是否打断当前 Host 运行 | 行为 |
|------|------|------------------------|------|
| **(1) 异常** | 超时、断链、节点失败、产物校验失败等 | **是**（门控） | 小鲸/系统整理现状 → 问卷 → 用户提交 → **续跑当前节点** |
| **(2) 会中请示** | `human_confirm: true`，主控/协作者判断需人审 | **是** | 会中产物 + 问卷 → 用户反馈 → Host **ReAct 继续**（下一轮 `run_current_node`） |
| **(3) 结果确认** | `human_confirm: true`，节点收敛完成 | **是** | `pending_delivery` + 总结 + 问卷（含 `decision`）→ 通过则归档并 **推进下一节点**；拒绝则返工 |
| **(4) 用户主动打字** | 对话框输入 | **否**（不打断正在跑的 Host） | 写入 `room_history`（会话历史可见）+ `user_context_pending`；**下次** `run_current_node` 注入 prompt |

### 3.1 表单提交（当前实现约定）

- 表单走 `intervene`，正文以 `[人工确认表单]` 开头。
- **非结果确认**（`pending_delivery.await_confirm != true`）：一律 **`resume_run=true`**，内容进入 `user_context_pending`，状态改 `processing`，调度 `schedule_run_node`。
- **结果确认**（`await_confirm=true`）：走 `confirm_node_delivery`（`approve`/`reject`）；拒绝时已 `schedule_run_node` 返工，**不**与「会中问卷续跑」混用。

### 3.2 用户打字（场景 4）

- `message_type=chat`：只记历史 + `user_context_pending`，**不** `resume_run`，**不**因聊天把 `human_intervention` 改成 `processing`。
- `message_type=instruction`（非结果确认门控）：在 `human_intervention` 下视为「继续处理」，**强制** `resume_run` + 注入上下文。

暂不支持「人为中断会议」专用流程，后续再加。

## 4. Host 与 Worker 协作（非会议室编排）

委派**不是** `rd_meeting` 逐步调度，而是 Host 在 **同一次** `execute_task_from_message` 的 ReAct 循环内调用工具：

```
run_current_node
  └─ Host.execute_task_from_message(节点 prompt + 用户上下文 + 返工意见)
       └─ ReAct 循环
            ├─ delegate_to_agent(worker_profile_id, message)  → await 子 Agent 完成 → 工具结果文本
            ├─ delegate_parallel([...])                       → 并行至多 5 路
            └─ 综合、检查、再委派…
       └─ 最终文本 report_body
  └─ 解析 hitl-questionnaire / 校验 / 门控 / 归档
```

| 要点 | 说明 |
|------|------|
| **同步等待** | `delegate_to_agent` 在工具返回前阻塞；Host 的「等待协作者」= 当前 tool_call 未结束 |
| **上下文隔离** | Worker 仅收到 `[任务背景][任务指令][委派原因]`，无 Host 全量会话历史 |
| **预热** | `rd_meeting:{room_id}:{worker_id}` 预先 `get_or_create`，委派走 `AgentOrchestrator.delegate` |
| **检查** | 会议 SKILL 要求 Host 对 Worker 产出做契合度/真实性/准确性检查后再采纳 |

子 Agent 进度（iteration、tools、tokens、`elapsed_s`）在 **聊天工作台** 通过 `Orchestrator._sub_agent_states` + WebSocket 可见；会议室 UI **已接** live 进度（`GET .../live` 3s 轮询 + `sub_agents` 字段渲染）。

## 5. 需求澄清节点（`req_clarify`）参考流程

默认：`human_confirm=true`（`default_human_confirm`），`NODE_INTENT` = 识别需求模糊点、交互式完善需求说明。

**建议 Host 技能（Agents 配置，非 binding）**

| 角色 | 技能 |
|------|------|
| Host | 需求澄清文档生成（模板）、`whalecloud-dev-tool-ask-user` |
| Worker 需求专家 | `whalecloud-dev-tool-requirement-clarify` 等 |
| Worker 代码专家 | 代码分析技能（知识体系偏代码） |

**开启 `human_confirm` 时的逻辑顺序（由 Host ReAct + 门控配合，非硬编码 FSM）**

1. 开会自动跑节点  
2. Host 分析能力卡片 + 会议目标，委派需求/代码 Worker  
3. 需要人审时输出 `hitl-questionnaire`（`kind=interactive`）→ 场景 (2)  
4. 用户表单/指令 → `user_context_pending` + `resume_run` → Host 继续 ReAct（再委派或自处理）  
5. 收敛后输出 `# 交付结论` + 总结 → 场景 (3) 问卷（`kind=result_confirm`）  
6. 用户确认通过 → 校验归档目录下智能体已写约定产物 → 推进下一节点（**禁止**流程代写覆盖）

**关闭 `human_confirm`**：无默认结果门控；异常仍须问卷；会中是否请示由 Host 判断（输出问卷标记）。

## 6. 环节提示词必备块

在 `build_node_prompt` + `build_room_skill_prompt` 中固定包含：

- 会议目标（`node_intent`）
- 协作智能体能力卡片（来自 Profile）
- `human_confirm` 开关及四类人工场景说明
- 工单信息（scope、标题；后续扩展描述字段）
- 系统信息（`WORK_ORDER_DIR`、`ARCHIVE_DIR`、`SYNAPSE_URL`、`GITNEXUS_URL` 等，`runtime_context`）
- 运营补充（`prompt_supplement`）

---

## 7. 数据字段

| 字段 | 位置 | 含义 |
|------|------|------|
| `user_context_pending` | `room_state.json` | 用户聊天/表单/指令队列，下次 `run_current_node` 注入后清空 |
| `pending_delivery` | `room_state` | 待确认总结；`await_confirm=true` 为结果确认门控 |
| `hitl_form_schema` | `room_state` | 当前门控问卷 schema |
| `rework_instruction` | `room_state` | 结果确认拒绝时的返工说明（单次消费） |
| `intervention_kind` | `room_state` | `interactive` / `result_confirm` / `exception` |

## 9. 相关 API

| API | 说明 |
|-----|------|
| `POST .../meeting-rooms/open` | 组装主控 prompt 后自动后台执行当前节点（需 `agent_pool`） |
| `POST .../meeting-rooms/{id}/intervene` | `text`、`message_type`（`instruction`/`chat`）、`resume_run`（表单/会中指令由服务端强制 true） |
| `GET .../meeting-rooms/{id}/live` | live 快照：phase、委派、子 Agent、`recent_chat`（轮询） |
| `POST .../meeting-rooms/{id}/reprocess` | 重新处理节点（当前节点或同阶段历史节点，清理过程数据后从 node_init 重跑） |
| `POST .../meeting-rooms/{id}/stop` | 终止当前节点后台运行，会议室标为 stopped |
| `POST .../meeting-rooms/{id}/review-decision` | NodeReviewPanel 三按钮入口：approve / reject / escalate |
| `GET .../meeting-rooms/{id}/node-review` | 读取节点确认总结 payload |
| `GET .../meeting-rooms/{id}/agent-trace` | 读取智能体节点级 trace（conversation/tools/skills/usage/events） |
| `GET .../meeting-rooms/{id}/artifact-file` | 读取归档文件原文（.md 内联展开用） |
| `GET .../meeting-rooms/{id}/nodes/{node_id}/participants` | 按 SOP 节点 binding 返回参会智能体阵容 |
| `GET .../meeting-rooms/{id}/agent-contexts` | 探测各参会 Agent 的 system prompt / messages（调试用） |
| `GET .../meeting-rooms/{id}/chat` | 按 SOP 节点读取协作会议流 chat_logs |
