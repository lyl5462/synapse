# whalecloud-dev-tool_his_order_query

## 技能目标
基于给定的需求单描述信息（`DEMAND_DESC`）和功能影响信息（`DEMAND_IMPACT`）作为主要检索依据，并结合功能特性（`PROD_FEATURE`）作为目标系统包含哪些功能模块的辅助理解材料，通过调用底层的图谱检索能力，精准查找到相似的历史工单单号列表，并提供有理有据的相似性证明说明。

## 核心输入参数 (Parameters)
该技能在被触发执行时，你必须通过环境变量或显式的参数形式接收以下上下文信息。**第一步必须是检查参数的完整性**。

| Parameter | 必填 | 说明 / 示例 |
|-----------|------|----------------|
| `SERVER_URL` | 是 | 提供底层图谱查询接口的服务器地址。你必须接收此参数，并在调用任何脚本时通过 `--server_url` 参数透传给脚本。例如："http://localhost:8000" |
| `DEMAND_DESC` | 是 | 当前需求单的自然语言描述及业务背景，例如："需要在CRM系统中新增客户流失预警功能，基于近三个月活跃度进行判断..." |
| `DEMAND_IMPACT` | 是 | 当前需求预估或已确认会影响的底层代码、功能模块、数据库表等范围信息，例如："影响模块：UserCenter；影响表：cust_active_log" |
| `PROD_FEATURE` | 是 | 说明目标系统包含哪些功能模块，作为辅助材料帮助你更好地理解 `DEMAND_DESC` 和 `DEMAND_IMPACT` 的业务上下文，例如："系统包含：客户管理、订单中心、流失预警等模块" |
| `PROD` | 否 | 产品标识，用于过滤特定产品的图谱数据（如果未提供，图谱查询脚本中默认不按产品强过滤或需由大模型根据上下文推断）。例如："CRM" |
| `DEBUG` | 否 | 调试模式开关。如果显式传入 `true`，你需将每一步的执行日志输出到调试文件中。默认 `false` |

## 工作流

### 步骤零：参数完整性与调试初始化
1.  **校验参数**：首先检查 `SERVER_URL`、`DEMAND_DESC`、`DEMAND_IMPACT`、`PROD_FEATURE` 这四个必填参数是否均已提供。
2.  **异常处理**：如果有任何必填参数缺失，必须立即停止后续流程，并向用户提示缺失的具体参数名称（如 SERVER_URL、DEMAND_DESC 等），要求补充以便进行准确的检索。
3.  **调试文件初始化**：如果接收到 `DEBUG=true`，需创建一个调试文件（例如 `docs/his_order_query_debug.log` 或当前工作目录下的 `.log` 文件），并在后续所有步骤中，将关键的思考过程、调用的脚本命令、传入的参数及脚本返回的原始结果，通过文件写入工具追加到该文件中。

### 步骤一：语义与特性的初步召回
1.  **意图提炼**：以 `PROD_FEATURE` 为业务上下文辅助理解，从 `DEMAND_DESC` 和 `DEMAND_IMPACT` 中准确提炼出用于检索的核心搜索词。
2.  **执行检索**：调用 `hybrid_query.py` 脚本，获取初步的相似历史需求节点列表（候选集），提取这些节点的 UUID 和单号。
    - 参考：[references/hybrid_query.md](references/hybrid_query.md)
3.  *(DEBUG)*：若 `DEBUG=true`，需将“提取出的核心搜索词”、“调用 `hybrid_query.py` 的完整命令参数”以及“初步召回的 UUID 和单号列表（含相似度得分）”写入调试日志。

### 步骤二：影响范围的拓扑交叉验证
1.  **执行展开**：针对步骤一获取的高置信度历史工单，遍历其 UUID，调用 `relation_query.py` 脚本进行拓扑检索。
    - 参考：[references/relation_query.md](references/relation_query.md)
2.  **重合度计算**：获取历史工单实际关联的模块、代码文件等拓扑数据，将其与用户输入的 `DEMAND_IMPACT` 信息进行对比，筛选出在影响范围上真实存在重合的工单，剔除“描述相似但改动完全不同”的干扰项。
3.  *(DEBUG)*：若 `DEBUG=true`，需将“为哪些 UUID 执行了拓扑展开”、“调用 `relation_query.py` 的命令”、“实际查出的影响模块/代码列表”以及“与输入 `DEMAND_IMPACT` 交叉对比匹配和剔除的详细判定过程”写入调试日志。

### 步骤三：深度关联检索（进阶使用）
1.  **高级过滤**：如果常规检索无法快速缩小范围，将主动构造 Cypher 语句，调用 `cypher_query.py`。利用对 `PROD_FEATURE` 模块背景的理解，结合 `DEMAND_DESC` 映射到图谱中的特定节点，例如：通过图谱直接过滤出业务描述匹配，且存在边指向特定 `Impact` 节点的工单，实现“一击必中”。
    - 参考：[references/cypher_query.md](references/cypher_query.md)
2.  *(DEBUG)*：若 `DEBUG=true`，需将“决定使用 Cypher 的推理理由”、“构造出的原始 Cypher 查询语句”、“传入的 `parameters`”以及“Cypher 返回的精确结果集”写入调试日志。

### 步骤四：结果整合与输出
1.  **数据汇总**：整合前三步的数据结果，锁定最匹配的历史工单。
2.  **相似度判定方法（判定标准）**：
    在评估历史需求单与当前需求单的相似度时，必须严格基于以下两个核心维度的权重进行打分或综合判定：
    *   **业务语义相似度 (权重 50%)**：
        *   **判定依据**：在 `PROD_FEATURE` 提供的系统模块背景下，当前 `DEMAND_DESC` 与历史工单的标题、描述是否存在业务意图、应用场景或解决痛点上的重合。
        *   **证明来源**：混合检索 (`hybrid_query.py`) 返回的结果排序。
    *   **底层影响重叠度 (权重 50%，一票否决权)**：
        *   **判定依据**：在 `PROD_FEATURE` 提供的上下文辅助下，当前需求预估的 `DEMAND_IMPACT`（如影响的模块、代码、数据库）与历史工单在实际开发落地时（`(:Demand)-[:SPLIT_INTO]->(:Task)` 关联的任务单）所产生的影响是否发生实质性重叠。
        *   **特别说明**：如果两个需求业务描述极其相似，但底层修改的模块和架构完全无关（重叠度为 0），则应判定为**不相似（伪相似）**。
        *   **证明来源**：拓扑展开 (`relation_query.py`) 或 Cypher 查询 (`cypher_query.py`) 得到的影响面证据。
3.  *(DEBUG)*：若 `DEBUG=true`，需将“每一个候选工单在两个维度的分别打分情况”、“一票否决的具体应用实例（如果有）”以及“最终选定并准备输出给用户的论证草稿”写入调试日志。
4.  **输出规范**：
    *   **相似历史工单单号列表**：清晰列出相似的历史工单号（如：`REQ-2023-001`）。
    *   **相似性的证明说明**：必须包含以下两个维度的论证事实：
        1.  **业务语义层面**：结合 `PROD_FEATURE` 提供的系统背景，解释历史工单与当前 `DEMAND_DESC` 在业务意图和解决场景上的匹配点。
        2.  **底层影响层面**：基于 `relation_query.py` 检索到的拓扑证据，指明历史工单与当前 `DEMAND_IMPACT` 在代码/模块/架构层面修改的实质性重叠部分。
